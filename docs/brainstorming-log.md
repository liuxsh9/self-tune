# Self-tune Brainstorming Log

> 从零开始的完整思考记录。记录了每一个创意的来源、讨论过程和决策理由。

## 1. 初始想法（用户提出）

**核心观察：** Claude Code + Sonnet 在帮助用户研发时，经常需要反复调试、优化，有时需要用户提醒和帮助发散思维才能正确完成工作。这样的"纠正/用户指导的记录"是值得学习的信息。

**与现有思路的区别：** 不同于 SWE 数据蒸馏（受沙箱环境和 test case 约束），这更像是在和用户交互中提升的场景，需要自主确认哪次交互是有价值的。

**初始设想的学习方式：** 构建 SFT 数据，包括：
- 针对失败/思维局限的点凝练成的 query
- 基于此次经验所形成的更优 CoT 和 response

**关键细节（用户举例）：** 比如代理网络不稳定的问题，最后用户换了代理就好了。此时关键经验应该是"主动测试代理连通性，不稳定时提示用户"，而不是"应该换到某个固定端口"。这个例子奠定了**泛化 vs 具体**的核心设计思考。

## 2. 需求澄清（逐步问答）

### 2.1 数据来源 → 多源混合
初期聚焦 Claude Code 对话日志，未来扩展到 Cursor、Copilot 等。

### 2.2 价值判定 → LLM 自我反思判定
不依赖启发式规则或用户手动标记，由模型自己做反思判定。对 Judge Model 能力要求高。

### 2.3 SFT 数据粒度 → 决策点级别（细粒度）
从一段长对话中精准识别多个"认知转折点"，每个转折点独立生成 SFT 数据。

### 2.4 泛化策略 → 双版本保留
同时保留具体上下文版本和泛化抽象版本，根据训练需求选用。

### 2.5 用户反馈：query 的设计理念
**用户补充的关键想法：** query 应提供尽可能像真实轨迹中大模型输入的"上文"，像"事故回放现场"。CoT 和 response 则比当时关键异常点做出的思考和决策更优。

→ 本质是 **counterfactual trajectory optimization**：在真实失败场景下，构建"如果当时思路正确，应该怎么想、怎么做"。

### 2.6 目标用户 → 个人/团队 → 公司级
最初定位个人使用，讨论中升级为公司级可部署模式，支持多用户数据汇聚。

### 2.7 用户的三个关键约束
1. **用户不一定是对的** — 不能盲目将修正当 ground truth
2. **个人喜好 vs 通用知识** — 偏好修正应转化为"问询元知识"
3. **通用经验需要深度判定** — 什么值得所有人学习需要严肃判定

### 2.8 技术栈 → 用户让我决定，并提出关键反馈
**用户的反馈改变了架构方向：** "为何不考虑做成 skill？不希望侵入式修改脚手架，更希望复用 Claude Code + Sonnet 完成经验总结。"

→ 这个反馈直接将架构从"外部重型系统"转向"Skill-Native 轻量架构"。

### 2.9 实现形态 → Skill + 离线工具 + Web UI
形态 B+ 被选中，但用户补充：不限于个人使用，要构建公司级可部署模式。

### 2.10 数据安全 → 内部可信环境
本地不做脱敏，中央服务器后处理阶段再进行安全和敏感信息处理。

### 2.11 Judge 模型策略 → 分层
实时反思用当前模型（快速免费），中央服务器用强模型做二次筛选和质量提升。

## 3. 方案选择

### 三个候选方案
1. **Skill-Native 架构**（推荐，被选中）— Skill 为核心，渐进式演进
2. Hook-Driven 被动采集 + 外部分析 — Hook 获取上下文有限
3. CLAUDE.md 指令 + 自定义存储 — 不可靠，不适合公司级

### 选择理由
- Skill 在对话上下文中运行，天然获取完整交互轨迹
- 零基础设施依赖，安装 skill 就能用
- 复用 Claude Code + Sonnet 的能力
- 渐进式路径清晰：Skill → CLI → 中央服务器

## 4. 核心概念模型设计

### 4.1 概念定义
- **Trace（轨迹）**：一次完整会话记录
- **Episode（片段）**：Trace 中的一个"认知转折点"
- **Insight（洞察）**：从 Episode 提炼的经验（双版本：concrete + abstract）
- **SFT Sample（训练样本）**：query + cot + response
- **Reminder（强提醒）**：可写入 CLAUDE.md 的经验规则

### 4.2 Insight 分类（insight_type）
- `skill_gap` — 模型能力缺陷
- `knowledge_gap` — 模型知识盲区
- `reasoning_error` — 推理/决策偏差
- `preference_probe` — 用户偏好类（转化为问询模式）
- `env_specific` — 环境特定问题
- `exploration_inefficiency` — 探索路径低效（v1 新增）
- `tool_orchestration` — 工具选择/编排不当（v1 新增）
- `backtrack_failure` — 未能及时放弃错误方向（v1 新增）

### 4.3 双产物飞轮（用户提出的核心创意）
**用户的关键想法：** 除了 SFT 数据（长期），还应该有"副产物"——类似 CLAUDE.md 强提醒（短期）。

```
同一条经验提炼流程
    ├→ 长期：SFT 数据 → 训练 → 提升模型根能力（惠及所有用户）
    └→ 短期：强提醒 → 写入 CLAUDE.md → 立刻提升当前用户体验
```

**飞轮效应：**
- 强提醒让用户立刻受益，验证经验是否有用
- 用户对提醒的保留/删除行为，反过来成为 SFT 数据质量信号

## 5. 触发机制设计

### 5.1 Skill 能力边界研究
- Skill 不能自动触发自己 — 必须由模型主动调用 Skill tool
- Skill 可以 spawn subagent — 通过 Agent tool
- CLAUDE.md 指令优先级最高 — 可引导模型行为
- Superpowers 的 `using-superpowers` meta-skill 机制可以复用

### 5.2 用户的关键反馈改变了触发设计
**用户指出：** 不能依赖手动触发。用户不会主动去总结经验，也不想花等待时间。
**用户提出两种自动触发时机：**
1. 过程中实时触发（任务由失败到成功时），用 sidecar 不打断主工作流
2. 任务完成后全过程回顾

### 5.3 最终触发方案
**核心发现（来自用户提问）：** superpowers 的 brainstorming skill 不是手动触发的，而是模型通过 skill description 自动匹配的。Self-tune 完全可以复用这个机制。

→ CLAUDE.md 中只需极少配置，触发逻辑由 Skill description 驱动自动匹配。

### 5.4 三种触发时机
- **Sidecar 模式**：过程中检测到转折点 → spawn background agent → 主工作流不阻塞
- **Retrospective 模式**：任务完成后 → background agent 做全过程回顾
- **Manual 模式**：用户主动调用 /self-tune（保留但不依赖）

### 5.5 失败任务和错误修正处理
**用户的关键补充：** "最后失败了也不代表中间的经验都是没价值的"

- 每个 Episode 独立评估价值，不因任务整体结果全盘否定
- 事后发现错误时：标记原 Insight 为 invalidated，生成新 Insight，更新 Reminder
- Reminder 有完整的生命周期管理（validation_count, contradiction_count）

## 6. 上下文隔离设计

### 6.1 用户提出的核心约束
**用户的担心：** 反思过程可能让模型上下文爆炸，或者污染用户的工作上下文。

### 6.2 第一设计原则
**绝不降级用户的原有工作体验。** 所有反思工作在 subagent 中执行，主对话 context 影响 < 100 tokens。

### 6.3 Subagent 接收精心构造的上下文包
不传完整对话历史，只传：任务描述、Episode 摘要、项目上下文。Subagent 启动时 context 很小，有足够空间做推理。

## 7. SFT 数据 Query 设计（多轮重要讨论）

### 7.1 第一次修订：保留长上下文
**用户的洞察：** 如果 query 变成简单问题，模型无法将学到的模式映射到真实 agentic 场景。SFT 数据的 query 应该反映真实工作场景的上文分布。

### 7.2 第二次修订：assistant ↔ tool 交互为主
**用户的关键修正：** "实际的交互中，是不是 assistant 和 tool 比较多？用户参与度不会那么高。"

真实分布：用户说了 2 句话，assistant + tool 交互了 10+ 轮。query 必须反映这个真实分布。

### 7.3 第三次修订：不包含用户提示（最重要的设计决策之一）
**用户的核心洞察：** "之前需要 user 来指导，但 SFT 后模型能自主更优思考。所以 SFT 数据不建议包含 user 的提示。"

→ 用户的提示是学习信号的来源，但不应出现在训练数据的 query 里。SFT 的目标是让模型内化这些智慧，实现自主推理。

这引出了 Subagent 的核心能力：**提示内化（Prompt Internalization）**

### 7.4 第四次修订：压缩 assistant ↔ tool 的低效交互
**用户进一步延伸：** 不只是内化用户提示，更重要的是优化 assistant ↔ tool 自身的交互效率。比如 10 轮才发现的问题，SFT 后 5 轮就能发现。

→ 引出 **最优切点识别（Optimal Decision Point Detection）**：T_optimal vs T_actual

### 7.5 最终的 SFT 数据来源矩阵

| 来源 | query 切点 | CoT 核心改造 |
|------|-----------|-------------|
| 类型1: 内化用户提示 | T_actual 前 | 用户智慧→自主推理 |
| 类型2: 压缩探索路径 | T_optimal | 冗长探索→最优路径 |
| 类型3: 修正错误经验 | 发现矛盾时 | 旧认知→新认知 |
| 类型4: 偏好→问询转化 | 偏好出现前 | 自行决定→主动询问 |
| 类型5: 放弃回退决策 | 应放弃的时刻 | 坚持错误方向→及时转向 |
| 类型6: 工具编排优化 | 低效tool调用前 | 低效序列→高效序列 |

## 8. 专家评审（三个并行 Subagent）

### 8.1 Skill 专家评审
**关键发现：**
- Description 不能包含 workflow 描述（Skill 规范的最严格反模式）
- 应使用即时性触发条件（"just received", "about to", "realizes"）
- 需要与 systematic-debugging 等 skill 明确排除条件
- CLAUDE.md 写入必须有 approval gate

### 8.2 SFT 训练专家
**新增类型（纳入 v1）：**
- 类型 5: 放弃回退决策
- 类型 6: 工具编排优化
- 类型 7: 信息充分性判断（合并到类型 2）

**CoT 质量控制：**
- 必须避免：事后合理化、无信息量谨慎、过度解释基础知识
- 应该强化：证据链推理、决策树外化、预期-观察-修正循环

**前沿方法借鉴：**
- DPO 数据可以从已有的 (低效轨迹, 优化轨迹) 对中免费获得
- Curriculum Learning 式数据排序提升训练效率

### 8.3 Agentic 工作流专家
**模型典型失败模式：**
- 鸵鸟调试（修表象不修根因）
- 散弹枪修改（不理解变更波及范围）
- 隧道视野（在错误方向越走越深）
- 复制粘贴编程（不理解代码库约定）

**成功交互也值得学习：**
- 精准的初始侦察（花 3 步省 30 步）
- 优雅的多文件协调修改
- 创造性的工具组合
- 防御性编程意识
→ 标记为 "expert trajectory"，高权重训练（Phase 2）

### 8.4 数据质量专家
**核心机制（纳入 v1）：**
- 对抗性反思：生成对立归因 + 置信度打分，解决"用户不一定对"
- 泛化阶梯：L1/L2/L3 三层表述，默认用最具体的 L1
- CoT 反模式过滤规则

**核心机制（Phase 2-3）：**
- 代际隔离（防止 model collapse）
- 用户可信度评分
- 领域感知采样
- 跨模型投票验证

**务实建议：** 先采集 1-2 个月数据，做离线分析后再开始训练。
